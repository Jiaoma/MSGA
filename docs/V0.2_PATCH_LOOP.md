# MSGA v0.2 设计文档：Verified Patch Loop

> 版本主题：可验证最小修复循环（Verified Patch Loop）  
> CLI 入口：`msga patch`  
> 设计日期：2026-07-19  
> 研究依据：2025–2026 年小语言模型编码代理、工具调用、自我修正与长上下文外部化研究综述

---

## 1. 背景与目标

MSGA 的项目目标是让本地小语言模型（SLM, Small Language Model）成为可靠的软件工程代理。近期研究表明，SLM 在代理系统中并非不适合编码任务，而是不适合承担开放式、长上下文、无边界、无验证的自主修复。它更适合处理：

- 高频、低变化、边界清晰的子任务；
- 有外部验证信号的代码任务；
- 明确上下文和明确输出格式的局部修改；
- 可由工具、测试和状态机约束的受控 patch。

因此，v0.2 不追求“让小模型一次性完成复杂需求”，而是验证一个更具体、更可测的假设：

> 小模型只要被放进足够窄、可观测、可回滚、可验证的循环里，就可以稳定完成局部代码修复任务。

v0.2 的核心产品能力定义为：

> `msga patch` 在当前仓库中运行验证命令，若发现失败，则生成结构化失败报告，要求模型先声明 PatchIntent，再在安全边界内执行最小补丁，重新验证，直到成功或安全停止。

---

## 2. 非目标

v0.2 应刻意收窄范围，避免过早进入复杂多代理系统。

暂不做：

1. 完整 RAG / 向量数据库 / 全仓库语义检索；
2. 长期记忆系统；
3. 模型微调或训练链路；
4. 开放式多代理协商；
5. 远程 CI 集成；
6. 跨语言项目完整支持；
7. 大规模重构式自动修复；
8. 默认修改测试、配置或依赖。

v0.2 可以为未来这些能力采集 trace 数据，但不把它们作为本版本交付目标。

---

## 3. CLI 入口设计

### 3.1 基本命令

```bash
msga patch
```

默认含义：

1. 自动探测当前项目的验证命令；
2. 运行 baseline validation；
3. 若验证已通过，直接退出；
4. 若验证失败，进入 Verified Patch Loop；
5. 最多执行 3 轮 patch；
6. 成功则退出，失败则给出停止原因。

### 3.2 带目标描述

```bash
msga patch "修复当前 validator 测试失败"
```

`goal` 是诊断和补丁生成的上下文提示，但不能覆盖验证事实。若用户目标与实际失败不一致，系统应以验证结果为准，并在输出中说明差异。

### 3.3 指定验证命令

```bash
msga patch --check "npm test"
msga patch --check "npm test" --check "npm run build"
```

使用 `--check` 而非 `--test`，因为验证命令可以是 test、build、lint、typecheck 或自定义 smoke command。

### 3.4 控制修复轮数

```bash
msga patch --max-rounds 3
```

默认值建议为 3。超过 3 轮后，小模型很容易进入盲修状态，应停止并报告。

### 3.5 Dry run

```bash
msga patch --dry-run
```

只运行验证和诊断，生成 `ValidationResult`、`FailureReport` 和可能的 `PatchIntent`，但不实际修改文件。

### 3.6 权限开关

建议 v0.2 支持或预留：

```bash
msga patch --allow-test-edits
msga patch --allow-config-edits
msga patch --allow-dependency-edits
msga patch --non-interactive
msga patch --json
```

默认策略：

- 不允许自动修改测试；
- 不允许自动修改配置；
- 不允许自动修改依赖；
- 非交互模式下，需要确认时直接停止并返回结构化结果。

## 4. 总体流程

```text
msga patch
  ↓
Resolve validation commands
  ↓
Run baseline validation
  ↓
ValidationResult
  ↓
Rule-first FailureReport
  ↓
Model supplement, if needed
  ↓
PatchIntent
  ↓
PatchIntent Guard
  ↓
低风险自动执行 / 高风险确认 / 明确违规拒绝
  ↓
ProposedPatch(oldText/newText)
  ↓
Apply patch
  ↓
Diff Guard
  ↓
Re-run validation
  ↓
Regression Guard
  ↓
Success / Continue / Stop / Escalate
```

核心原则：

1. 模型不能直接改文件，必须先输出 PatchIntent；
2. FailureReport 的事实层由规则生成，模型只能补充解释；
3. PatchIntent 不能扩大 FailureReport 的允许范围；
4. 实际 diff 必须符合 PatchIntent；
5. 每轮 patch 后必须重新验证；
6. 一旦 regression 或连续无改善，应停止自动修复。

## 5. 状态机设计

```ts
type PatchLoopState =
  | "planning"
  | "validating"
  | "diagnosing"
  | "intent_reviewing"
  | "patching"
  | "diff_reviewing"
  | "revalidating"
  | "regression_checking"
  | "succeeded"
  | "failed"
  | "stopped";
```

每轮循环产生一条记录：

```ts
interface PatchLoopIteration {
  round: number;
  state: PatchLoopState;
  validationResult?: ValidationResult;
  failureReport?: FailureReport;
  patchIntent?: PatchIntent;
  patchIntentReview?: PatchIntentReview;
  proposedPatch?: ProposedPatch;
  diffGuard?: DiffGuardResult;
  regressionCheck?: RegressionCheck;
  changedFiles: string[];
  decision: "continue" | "stop" | "escalate" | "success";
}
```

## 6. ValidationResult

验证阶段负责运行命令并压缩基础输出。

```ts
interface ValidationResult {
  command: string;
  success: boolean;
  exitCode: number;
  durationMs: number;
  stdoutSummary: string;
  stderrSummary: string;
  rawStdout?: string;
  rawStderr?: string;
  failingFiles: string[];
  failingTests: string[];
  primaryError: string;
}
```

v0.2 优先支持：

- `npm test`
- `npm run build`
- `npm run lint`
- TypeScript `tsc`
- Vitest
- Node stack trace
- Biome / ESLint 基础格式

## 7. FailureReport：规则优先，模型补充

`FailureReport` 是 patch loop 的事实来源。原则是：

> 能用规则提取的事实，绝不交给模型猜；模型只能补充 suspectedCause、repairHint 和 riskNotes。

```ts
interface FailureReport {
  source: "rule" | "rule+model" | "model_fallback";

  failureType:
    | "test_failure"
    | "type_error"
    | "lint_error"
    | "build_error"
    | "runtime_error"
    | "command_error"
    | "unknown";

  command: string;
  exitCode: number;
  primaryError: string;

  failingTests: FailingTest[];
  diagnostics: Diagnostic[];
  evidence: FailureEvidence[];

  suspectedFiles: string[];
  allowedFilesToEdit: string[];
  disallowedFiles: string[];
  disallowedActions: string[];

  modelSupplement?: FailureModelSupplement;
  confidence: "high" | "medium" | "low";
}

interface FailingTest {
  name: string;
  file?: string;
  line?: number;
  message?: string;
}

interface Diagnostic {
  file?: string;
  line?: number;
  column?: number;
  code?: string;
  message: string;
  source: "typescript" | "vitest" | "jest" | "biome" | "eslint" | "node" | "npm" | "unknown";
}

interface FailureEvidence {
  kind: "stdout" | "stderr" | "stack" | "test" | "diagnostic";
  text: string;
}

interface FailureModelSupplement {
  suspectedCause: string;
  repairHint: string;
  riskNotes: string[];
}
```

### 7.1 规则诊断器管线

建议实现：

```ts
interface FailureAnalyzer {
  name: string;
  canAnalyze(result: ValidationResult): boolean;
  analyze(result: ValidationResult): Partial<FailureReport>;
}
```

按优先级运行：

```text
TypeScriptAnalyzer
VitestAnalyzer
JestAnalyzer
BiomeAnalyzer
ESLintAnalyzer
NpmAnalyzer
GenericStackTraceAnalyzer
FallbackAnalyzer
```

### 7.2 allowedFilesToEdit 推断规则

优先级：

1. TypeScript / lint diagnostic 中的源码文件；
2. stack trace 中的源码文件；
3. 测试文件 import 的源码文件；
4. 用户 goal 明确提到的源码文件；
5. 模型建议的候选源码文件。

过滤掉：

```text
tests/
*.test.ts
*.spec.ts
package.json
package-lock.json
tsconfig.json
.github/
node_modules/
dist/
coverage/
```

除非用户显式允许。

### 7.3 置信度规则

`high`：

- 有明确 TS/lint diagnostic 指向源码文件；或
- 测试失败 + stack trace 指向源码文件；或
- 测试失败 + import 关系唯一指向一个源码文件。

`medium`：

- 有测试失败；
- 能推断 2–3 个候选源码文件；
- 但没有明确行号。

`low`：

- 只有 command failed；
- 无文件、无测试名、无 stack；
- 需要模型猜测。

低置信度不应自动 patch。

## 8. PatchIntent-first 机制

模型不能直接进入改文件阶段，必须先声明 PatchIntent。

```ts
interface PatchIntent {
  targetFiles: string[];

  changeType:
    | "bug_fix"
    | "type_fix"
    | "test_fix"
    | "lint_fix"
    | "build_fix"
    | "dependency_fix"
    | "unknown";

  reason: string;
  expectedEffect: string;
  failureEvidence: string[];

  allowedOperations: PatchOperation[];
  forbiddenOperations: string[];

  riskLevel: "low" | "medium" | "high";

  maxChangedFiles: number;
  maxChangedLines?: number;
}

type PatchOperation =
  | "edit_function"
  | "add_guard"
  | "adjust_type"
  | "update_import"
  | "add_test"
  | "update_test"
  | "update_config"
  | "rename_symbol"
  | "small_refactor";
```

v0.2 默认安全操作：

```ts
const SAFE_OPERATIONS = [
  "edit_function",
  "add_guard",
  "adjust_type",
  "update_import",
];
```

默认不自动允许：

```ts
"add_test"
"update_test"
"update_config"
"rename_symbol"
"small_refactor"
"dependency_fix"
```

### 8.1 PatchIntent 与 FailureReport 的约束关系

```ts
PatchIntent.targetFiles ⊆ FailureReport.allowedFilesToEdit
PatchIntent.forbiddenOperations ⊇ FailureReport.disallowedActions
```

模型不能凭空扩大修复范围。

## 9. PatchIntent Guard

PatchIntent 审核结果分三类：

```ts
type PatchIntentDecision = "auto_approve" | "needs_confirmation" | "reject";

interface PatchIntentReview {
  decision: PatchIntentDecision;
  riskScore: number;
  reasons: string[];
  violations: string[];
  requiredFlags: Array<
    | "allow_test_edits"
    | "allow_config_edits"
    | "allow_dependency_edits"
  >;
  normalizedIntent?: PatchIntent;
}
```

### 9.1 低风险自动执行

全部满足时自动执行：

1. `FailureReport.confidence` 为 `high` 或 `medium`；
2. `PatchIntent.riskLevel === "low"`；
3. `targetFiles.length <= 2`；
4. 所有 `targetFiles` 都在 `allowedFilesToEdit` 内；
5. 不修改测试文件；
6. 不修改配置文件；
7. 不修改依赖文件；
8. `allowedOperations` 仅包含安全操作；
9. `maxChangedLines <= 40`；
10. 没有危险意图关键词，如 rewrite、disable、skip test 等。

### 9.2 需要确认

以下情况进入确认：

- 诊断置信度低；
- `riskLevel === "medium"`；
- 涉及测试文件；
- 涉及配置文件；
- 涉及依赖变更；
- 修改 3 个左右文件；
- `maxChangedLines > 40`；
- 包含 `add_test`、`update_test`、`small_refactor`、`rename_symbol` 等操作。

### 9.3 必须拒绝

以下情况直接拒绝：

- 目标文件越界；
- 目标文件不存在；
- 试图修改 `node_modules/`、`dist/`、`coverage/`；
- 试图绕过验证；
- 高风险大重构；
- 修改依赖但未显式允许；
- 出现危险代码或测试模式。

危险模式示例：

```text
.skip(
.only(
assert(true)
expect(true).toBe(true)
process.exit(0)
noEmitOnError: false
```

## 10. ProposedPatch：oldText/newText 精确替换

v0.2 不建议要求 SLM 输出 unified diff。更适合采用精确文本替换：

```ts
interface ProposedPatch {
  edits: TextEdit[];
}

interface TextEdit {
  file: string;
  oldText: string;
  newText: string;
  reason: string;
}
```

系统应用 patch 时必须检查：

1. `file` 属于 `PatchIntent.targetFiles`；
2. `oldText` 在文件中唯一匹配；
3. `newText !== oldText`；
4. 修改后文件仍可读；
5. 实际变更不超过 PatchIntent 限制。

若 `oldText` 不唯一或不存在，应拒绝该 patch，不应让模型自由重试大范围改写。

---

## 11. Diff Guard

patch 应用后必须审查实际 diff。

```ts
interface DiffGuardResult {
  approved: boolean;
  changedFiles: string[];
  addedLines: number;
  deletedLines: number;
  violations: string[];
}
```

检查规则：

1. 实际修改文件必须属于 `PatchIntent.targetFiles`；
2. changed file 数量不能超过 `maxChangedFiles`；
3. changed lines 不能超过 `maxChangedLines`；
4. 不得修改测试、配置、依赖，除非显式允许；
5. 不得删除大量代码；
6. 不得出现绕过验证的危险模式。

---

## 12. Regression Guard

每轮 patch 后重新运行验证，并判断是否真的变好。

```ts
interface RegressionCheck {
  previousFailingCount: number;
  currentFailingCount: number;
  newFailures: string[];
  fixedFailures: string[];
  changedFilesDelta: string[];
  verdict: "improved" | "unchanged" | "regressed";
}
```

停止条件：

1. 验证通过：成功退出；
2. 新增失败：停止；
3. 连续 2 轮无改善：停止；
4. 修改范围扩大：停止；
5. 同一 primary error 连续出现：停止；
6. patch intent 与实际 diff 不一致：停止；
7. 达到 `maxRounds`：停止。

失败时应输出清晰原因，而不是继续盲修。

## 14. Trace 数据

每次 `msga patch` 都应保存 trace，为后续研究、评测和微调数据采集做准备。

```ts
interface PatchTrace {
  goal?: string;
  checks: string[];
  baselineValidation: ValidationResult;

  iterations: Array<{
    round: number;
    failureReport: FailureReport;
    patchIntent?: PatchIntent;
    patchIntentReview?: PatchIntentReview;
    proposedPatch?: ProposedPatch;
    diffGuard?: DiffGuardResult;
    validationAfterPatch?: ValidationResult;
    regressionCheck?: RegressionCheck;
    decision: "success" | "continue" | "stop" | "escalate";
  }>;

  finalOutcome: "success" | "failed" | "stopped" | "regressed";
}
```

这些数据未来可以回答：

- PatchIntent 合规率是多少？
- oldText/newText 应用失败率是多少？
- 常见越界意图是什么？
- 多少失败来自诊断错误，多少来自 patch 错误？
- 哪类任务需要升级到更大模型？

```text
  rerun with a narrower goal or inspect src/core/validator.ts manually
```

## 16. 实现切片

### Slice 1：CLI skeleton

实现：

```bash
msga patch
msga patch "goal"
msga patch --check "npm test"
msga patch --max-rounds 3
msga patch --dry-run
```

此阶段可只解析参数并打印 patch config。

### Slice 2：Baseline validation

实现统一入口：

```ts
runValidation(commands: string[]): Promise<ValidationResult[]>
```

支持命令执行、exit code、stdout/stderr summary、durationMs。

### Slice 3：FailureReport 规则压缩

实现 Analyzer 管线：

- TypeScriptAnalyzer
- VitestAnalyzer
- NpmAnalyzer
- GenericStackTraceAnalyzer
- BiomeAnalyzer / ESLintAnalyzer
- FallbackAnalyzer

### Slice 4：Dry-run 诊断

`msga patch --dry-run` 输出：

- ValidationResult；
- FailureReport；
- allowedFilesToEdit；
- disallowedActions；
- 若触发模型补充，则包含 modelSupplement。

### Slice 5：PatchIntent-first

接入模型生成 PatchIntent，并实现 PatchIntent Guard。

### Slice 6：ProposedPatch 应用

实现 oldText/newText 精确替换，拒绝非唯一匹配或越界文件。

### Slice 7：Diff Guard + Regression Guard

重新验证，比较失败数量、新增失败、修改范围和连续无改善轮数。

### Slice 8：Trace 保存

保存 `.msga/traces/patch-*.json`。

## 17. MVP 验收标准

v0.2 MVP 完成时应满足：

1. `msga patch --check "npm test"` 能运行验证命令；
2. 验证失败时能生成结构化 `ValidationResult`；
3. 能从 TypeScript/Vitest/npm 输出中生成 `FailureReport`；
4. 能推断 `allowedFilesToEdit`，且默认排除测试和配置；
5. 模型必须先输出 PatchIntent；
6. 低风险 PatchIntent 自动执行，高风险进入确认或停止；
7. patch 使用 oldText/newText 精确替换；
8. patch 后执行 Diff Guard；
9. patch 后重新运行验证；
10. regression 或连续无改善时停止；
11. 每次运行保存 PatchTrace；
12. 文档和 CLI help 明确说明默认安全边界。

---

## 18. 一句话版本定义

> MSGA v0.2 实现 `msga patch` 可验证最小修复循环，通过规则优先的失败报告、PatchIntent-first 的受控补丁生成、低风险自动执行/高风险确认策略，以及 Diff Guard 与 Regression Guard，让小语言模型能够在有限上下文和有限权限内可靠修复局部代码问题。
